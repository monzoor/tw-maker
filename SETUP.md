# Project Setup — One-Time Initialization

> **Use this file once to initialize a new project from scratch.**
> After setup is complete, this file is no longer needed for day-to-day development.
> All architecture rules and conventions live in `.github/copilot-instructions.md`.

---

## Bootstrap — Initialize the Project from Scratch

Run these commands in order. Do not skip any step.

```bash
# 1. Initialize the Next.js app in the current folder.
cd <project-name>
mkdir -p /tmp/setup-stash && mv .github SETUP.md /tmp/setup-stash/
bun create next-app@latest . --yes --src-dir --experimental-test-proxy
mv /tmp/setup-stash/.github /tmp/setup-stash/SETUP.md .

cat node_modules/next/package.json | grep '"version"'   # confirm Next.js version
cat node_modules/react/package.json | grep '"version"'  # confirm React version

# 2. Install runtime dependencies
bun add @reduxjs/toolkit react-redux axios

# 3. Install dev dependencies
bun add -D @modelcontextprotocol/sdk ts-morph \
  figma-context-mcp mcp-design-system-extractor @danielsogl/lighthouse-mcp

# 4. Place .github/copilot-instructions.md in the project
mkdir -p .github

# 5. Create .vscode/mcp.json (full content below)
mkdir -p .vscode
```

### 6. Scaffold full directory structure

Run once immediately after the Next.js app is created to establish all directories the architecture depends on:

```bash
mkdir -p src/lib/redux/slices
mkdir -p src/lib/api
mkdir -p src/lib/services
mkdir -p src/lib/types
mkdir -p src/lib/analytics
mkdir -p src/lib/seo

mkdir -p src/app/styles
mkdir -p src/app/api
mkdir -p src/mcp/tools
mkdir -p src/scripts
mkdir -p src/tokens
```

Also create `global.d.ts` at the project root and add it to `tsconfig.json` `include` for ambient `NodeJS.ProcessEnv` declarations.

`src/proxy.ts` is auto-generated by the `--experimental-test-proxy` flag. Replace its contents with the auth guard skeleton from `.github/copilot-instructions.md`. Confirm the export name and `config.matcher` against `node_modules/next/dist/docs/01-app/01-getting-started/16-proxy.md` before editing.

Add `typedRoutes: true` to `next.config.ts` — this is a stable option (not experimental) that makes Next.js validate all `href`/`push`/`replace` strings at compile time. It is **not** enabled by `create-next-app` and must be set manually. Confirm the option name against `node_modules/next/dist/docs/01-app/03-api-reference/05-config/01-next-config-js/typedRoutes.md`.

## Project Plan (Implementation Roadmap)

### Phase 1 — Token Schema & CSS Pipeline

1. Create `src/tokens/tokens.json` — single source of truth for all design tokens:
   - `colors`: brand, semantic (success/warning/error/info), neutral scale
   - `typography`: fontFamily, fontSize, fontWeight, lineHeight, letterSpacing
   - `spacing`: 0–96 scale
   - `borderRadius`
   - `shadows` (elevation levels)
   - `breakpoints` (screen sizes)
2. Create `src/scripts/generate-tokens-css.ts` — reads `src/tokens/tokens.json`, outputs `src/app/styles/tokens.css` with a `@theme { ... }` block using Tailwind v4 CSS variable namespaces: `--color-*`, `--font-*`, `--text-*`, `--leading-*`, `--tracking-*`, `--spacing-*`, `--radius-*`, `--shadow-*`, `--breakpoint-*`
3. Update `src/app/styles/globals.css` — Tailwind v4 does **not** use `@tailwind base/components/utilities` (those are v3 only and will error in v4). The correct v4 content is:
   ```css
   @import "tailwindcss";
   @import "./tokens.css";
   ```
   Remove any `@tailwind base`, `@tailwind components`, or `@tailwind utilities` lines if present.
4. Add `"tokens:generate": "bun run src/scripts/generate-tokens-css.ts"` to `package.json`

**Token workflow:** Edit `src/tokens/tokens.json` → run `bun run tokens:generate` → Tailwind picks up the new CSS variables automatically.

#### 5a — Define `tokens.json` first (do this before any component)

Fill in all token categories in `src/tokens/tokens.json` — colors (brand, semantic, neutrals), typography (fontFamily, fontSize, fontWeight, lineHeight), spacing scale, borderRadius, shadows, and breakpoints. Run `bun run tokens:generate` and verify `src/app/styles/tokens.css` is produced with correct `@theme` variables. **Do not start 5b until this is done** — token choices made here propagate to every component automatically.

#### 5b — Build the reference atom: `Button`

Implement `src/components/atoms/Button/Button.tsx` fully — all variants (primary, secondary, ghost, destructive) and all sizes (sm, md, lg). Use only Tailwind utilities driven by the CSS variables from `tokens.css` — no hardcoded color or spacing values. Write `Button.stories.tsx` immediately after with one story per variant. This file is the pattern every other component must follow for file structure, prop naming, and token usage.

> ⚠️ Stories require Storybook to be installed (Phase 3) before they are viewable. Write the files now — verify them visually after Phase 3 is complete.

#### 5c — Batch-generate all remaining atoms (~39 components)

Using the `Button.tsx` pattern as the reference, implement every remaining atom listed under `src/components/atoms/` in the Directory Structure tree in `.github/copilot-instructions.md`. The inline comment next to each file path describes the component — use it to determine variants, props, and behaviour. Write the `.stories.tsx` immediately after each `.tsx`. Work through the full atom list before moving to molecules.

**MVP fast-track:** If you want a working storefront sooner, prioritise these atoms first: `Input`, `Label`, `Badge`, `Icon`, `Avatar`, `Spinner`, `Skeleton`, `PriceTag`, `Heading`, `Text`, `Overlay`, `Logo`. The remaining atoms can follow.

#### 5d — Batch-generate molecules → organisms → templates

Repeat the same pattern level by level — always complete and review the current level before starting the next, since each level imports from the one below.

- **Molecules** — compose from the verified atoms. Each molecule's inline comment lists its composition (e.g. `FormField` = Label + Input + Text).
- **Organisms** — compose from verified molecules and atoms. For complex organisms (e.g. `CartDrawer`, `CheckoutStepper`, `SearchOverlay`), write a minimal default story only — add full variant stories after MVP ships.
- **Templates** — layout skeletons only; accept all content via `children` or slot props, no real data.

> ⚠️ Do not create `.stories.tsx` files before Phase 3 is complete and `bun run storybook` confirms Storybook is running. The files can exist on disk but cannot be verified until then.

### Phase 2 — Redux Toolkit

> Reference: [RTK + Next.js App Router guide](https://redux-toolkit.js.org/usage/nextjs). There is no built-in Next.js scaffold for Redux — create these files manually following the patterns in `.github/copilot-instructions.md`.

6. Create `src/lib/redux/store.ts` — `makeStore()` factory; infer and export `AppStore`, `RootState`, `AppDispatch` types
7. Create `src/lib/redux/hooks.ts` — export `useAppDispatch`, `useAppSelector`, `useAppStore` typed to `AppStore`
8. Create `src/lib/redux/StoreProvider.tsx` — `'use client'` provider with `useRef` guard so the store is created once per request
9. Create `src/lib/redux/slices/authSlice.ts` — `status: 'idle' | 'pending' | 'fulfilled' | 'rejected'`, `user`, `token`, `error`; `loginThunk` via `createAsyncThunk`; `reset` reducer; `selectUser`, `selectIsLoggedIn` selectors
10. Create `src/lib/redux/slices/cartSlice.ts` — `CartItem[]`, `status`, `error`; `itemAdded`, `itemRemoved` reducers; `reset` reducer; `selectCartItems`, `selectCartTotalCount` (memoized with `createSelector`) selectors
11. Create `src/lib/redux/slices/uiSlice.ts` — mobile menu open state, active modal, toast queue; `mobileMenuOpened`, `mobileMenuClosed` reducers; `reset` reducer; `selectIsMobileMenuOpen` selector
12. Wrap children in `src/app/layout.tsx` with `<StoreProvider>` imported from `@/lib/redux/StoreProvider`

### Phase 3 — Storybook 8

13. Install Storybook 8: run `bunx storybook@8 init` and select `@storybook/nextjs` framework when prompted. **Do not use `@latest` — it installs v10+ which is incompatible with the required addons.** Verify Storybook starts (`bun run storybook`) before writing any story files.
14. `.storybook/main.ts`: stories glob covers `components/atoms/**/*.stories.@(ts|tsx)`, `components/molecules/**/*.stories.@(ts|tsx)`, `components/organisms/**/*.stories.@(ts|tsx)`, `components/templates/**/*.stories.@(ts|tsx)`; addons: `@storybook/addon-essentials`, `@storybook/addon-a11y`, `@storybook/addon-themes`
15. `.storybook/preview.ts`: import `../src/app/styles/globals.css` so tokens are available in stories
16. Open Storybook in the browser and verify all `.stories.tsx` files written in steps 5b–5d are visible and render correctly. Fix any stories that render blank, throw an error, or fail to pick up token-based styles. Organisms with a `// TODO: story` comment can be skipped for now.

### Phase 4 — MCP Servers

Next.js 16 has built-in MCP support at `/_next/mcp` and ships a ready-made `next-devtools-mcp` package for runtime introspection. We run **six MCP servers** side by side.

#### 4a — Next.js DevTools MCP (built-in, no code required)

17. Install `next-devtools-mcp` as a dev dependency: `bun add -D next-devtools-mcp`
18. Add it to `.vscode/mcp.json` (already done above).
19. Start `bun run dev` — `next-devtools-mcp` auto-connects to `/_next/mcp` on the running dev server.

#### 4b — Design System MCP (custom)

20. Create `src/mcp/index.ts` — entry point using `StdioServerTransport`, registers all custom tools
21. Implement tools in `src/mcp/tools/`:
    - **`list_components`** — walks `components/` recursively, returns file paths + component names grouped by atomic level
    - **`read_component`** — takes a file path, returns full source code + structured props summary extracted via `ts-morph`
    - **`write_story`** — takes a file path + story content string, writes (or overwrites) the `.stories.tsx` co-located with the component
    - **`get_design_tokens`** — reads and returns `src/tokens/tokens.json` so agents have full token context when generating stories
22. Add `"mcp": "bun run src/mcp/index.ts"` to `package.json`

#### 4c — Storybook MCP (official)

23. Already registered in `.vscode/mcp.json` — no additional installation needed, runs via `npx`.

#### 4d — Figma Context MCP

24. Already installed (`figma-context-mcp`) and registered in `.vscode/mcp.json`. Requires `FIGMA_API_KEY` — VS Code will prompt on first use.

#### 4e — Design System Extractor MCP

25. Already installed (`mcp-design-system-extractor`) and registered in `.vscode/mcp.json`.

#### 4f — Lighthouse MCP

26. Already installed (`@danielsogl/lighthouse-mcp`) and registered in `.vscode/mcp.json`.
